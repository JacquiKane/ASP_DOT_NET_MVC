MVC, Design Patterns
Design that doesn’t take change into account risks major redesign - Erich Gamma question_answer

Design patterns are pre-made blueprints that you can customize to solve a design problem with code. The pattern is not a specific piece of code, but a general concept for solving a particular problem. You can follow the pattern details, guidelines and templates, and create your solution.

MVC is one such design pattern you can use to create coded solutions with a framework that can be tailored to suit your application.

The Model-View-Controller (MVC) pattern is an architectural pattern that separates an application into three main logical components Model, View, and Controller.

M stands for Model, the object that represents the data.

V stands for View, what the user sees.

C stands for Controller, handles the request and routes to the correct view, linking the correct model.

The MVC design pattern is much used in industry. is the key advantage of using MVC.

MVC frameworks exist for multiple languages, the pattern is the design concept and the framework is the starter structure and code.

🔑 The MVC pattern separates the user interface logic from the business logic in a dynamic web application. It is used extensively and currently, by software developers in the banking and services industry to design and build modern web applications.

In software engineering, separation of concerns is a design strategy for separating an application into distinct sections. Each section addresses a separate concern.

For example, all of the code that deals with what the user sees might be in one section, another section might deal with the data in the application, and yet another section might handle the requests from the user.

The Process

stars What's the sequence?

The user decides to take an action, like viewing details.

A part of the software called the controller then takes over.

The controller sends back the view, which is the markup, combined with the data from the database (if required).


You can alter the code and insert additional logic. You can customize the code base - the visuals and additional logic are up to the developer. However, the design pattern remains the same - Model View Controller.

The steps?

  • User requests an action, like viewing details of a basketball player
  • Controller software receives the request, gathers data and links the view
  • Controller software sends back the view and the data to the user

A connection to the database is important and this step has to be taken before the links are made between the Model representing the data, and the object(s) holding the data.

all work together to produce a result.
